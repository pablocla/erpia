## 1. Protheus vs ERP Argentina вҖ” modelo mental

| DimensГЈo | **Protheus (TOTVS)** | **ERP Argentina (este proyecto)** |
|----------|----------------------|-----------------------------------|
| **Lenguaje custom** | AdvPL вҖ” compilГЎs fuentes вҶ’ RPO en el AppServer del cliente | TypeScript вҖ” custom profundo = cambio en repo + deploy SaaS |
| **Diccionario de datos** | SIGACFG / metadatos en BD (campos, Гӯndices, triggers) sin tocar SQL a mano | Prisma + PostgreSQL: esquema en `schema.prisma` + migraciones SQL |
| **Campos nuevos вҖңsin programarвҖқ** | Alta en diccionario вҶ’ impacto en forms/informes estГЎndar | `CampoPersonalizado` + `ValorCampoPersonalizado` (EAV por empresa) + UI en `/dashboard/campos-personalizados` |
| **Activar mГіdulos / rutas** | ParГЎmetros MV_* + configuraciГіn por grupo/filial | **Railroad engine**: `FeatureRubro` вҶ’ `ConfiguracionRubro` вҶ’ `FeatureEmpresa` (`isFeatureActiva`) |
| **Flujos de negocio** | Puntos de entrada AdvPL + workflows locales | `WorkflowRubro` + `WorkflowEngine` (pasos, transiciones, logs en BD) |
| **CatГЎlogos propios del cliente** | Tablas SX* / cadastros | `TablaAuxiliar` + `ValorAuxiliar` |
| **Multi-empresa** | Filial (cГіdigo + tabela compartilhada) | Multi-tenant: `empresaId` en todas las queries (`whereEmpresa`) |
| **Escala horizontal** | VГЎrios AppServers + load balancer + TopConnect | Varias instancias Next.js stateless + PostgreSQL pool (Supabase/Railway) вҖ” **un deploy sirve N clientes** |
| **Escala vertical** | MГЎs CPU/RAM en AppServer y DB | MГЎs recursos en Postgres + instancia Node |
| **ActualizaciГіn** | Patch UPDDIST / pacote compilado por versiГіn | Hoy: `git pull` + `prisma migrate deploy` + `npm run build` вҖ” **manual, sin interfaz de parches** |
| **PersonalizaciГіn вҖңestГЎndarвҖқ** | SeparaciГіn fuente custom vs estГЎndar (merge en upgrade) | SeparaciГіn por capas (ver В§3); riesgo si se modifica core sin extension hooks |

**ConclusiГіn:** Este ERP es **SaaS multi-tenant con parametrizaciГіn en BD**, no un ERP вҖңcompilable en sitioвҖқ como Protheus. La escalabilidad horizontal es por **rГ©plicas de la misma app**; la verticalizaciГіn es por **rubro + features + workflows**, no por un binario AdvPL por cliente.

### Capa 1 вҖ” ConfiguraciГіn sin tocar BD estructural
- Plan de cuentas, numeradores, parГЎmetros fiscales, listas de precio, usuarios/roles.
- **DГіnde:** `/dashboard/configuracion`, onboarding por rubro.
- **Analista:** capacitaciГіn + datos maestros.

### Capa 2 вҖ” Equivalente parcial a SIGACFG (campos y tablas)
- **Campos personalizados:** `CampoPersonalizado` por `entidad` (cliente, producto, facturaвҖҰ).
- **Tablas auxiliares:** catГЎlogos (вҖңOrigen leadвҖқ, вҖңTipo envГӯoвҖқ).
- **API:** `/api/campos-personalizados`, `/api/config/tablas-auxiliares`.
- **LГӯmite:** no crean joins automГЎticos en reportes complejos; hay que cablear en pantallas que lean `ValorCampoPersonalizado`.

### Capa 3 вҖ” Railroad / features (equivalente a вҖңactivar mГіdulo MVвҖқ)
- Activar KDS, picking, agro, membresГӯas por empresa.
- ParГЎmetros JSON por feature (ej. dГӯas cobranza IA).
- **CГіdigo:** `lib/config/rubro-config-service.ts`, `configuracion-feature-service.ts`.
- **Doc tГ©cnica:** `docs/funcional/rubro-config-railroad-engine.md`.

### Capa 4 вҖ” Workflows
- Secuencias por rubro: venta вҶ’ picking вҶ’ remito вҶ’ factura.
- **CГіdigo:** `lib/config/workflow-engine.ts` (colas de aprobaciГіn aГәn TODO BUG-004).

### Capa 5 вҖ” Desarrollo a medida
- Nueva pantalla, nueva regla fiscal, integraciГіn ML/ERP legado.
- Requiere PR, tests (`__tests__/`), revisiГіn multi-tenant.

## 5. Actualizaciones вҖ” estado actual y modelo objetivo

### Hoy (manual вҖ” вҖңmodo Protheus sin UPDDISTвҖқ)

1. Mantenedor publica versiГіn en Git (tag `vX.Y.Z`).
2. Por cada entorno (staging/prod):
   - `git pull` / deploy Vercel-Railway
   - `npx prisma migrate deploy`
   - `npm run build` + restart
3. **Todos los tenants** de ese entorno reciben el mismo cГіdigo al mismo tiempo.
4. Personalizaciones capa 2вҖ“3 **persisten** en BD; capa 5 custom puede **romperse** si no hay tests.

**No existe aГәn:**
- Interfaz de parches tipo UPDDIST
- ActualizaciГіn por tenant aislado
- Merge automГЎtico custom vs estГЎndar

### Objetivo recomendado (roadmap producto)

| Modo | CuГЎndo | Mecanismo |
|------|--------|-----------|
| **SaaS rolling** | Clientes en cloud compartido | Deploy blue/green; migraciones backward-compatible; feature flags para activar cambios |
| **Tenant schema drift** | Solo campos EAV | Sin migraciГіn; ya soportado |
| **Parche SQL** | Cambio estructural BD | `prisma/migrations/*` versionado; nunca SQL ad hoc en prod |
| **Parche config** | Nuevas features por rubro | Seed idempotente `configuracion-feature-service` + script `apply` por empresa |
| **Hotfix cliente Гәnico** | Enterprise | Rama + deploy dedicado O override capa 3вҖ“4 (evitar capa 5) |
| **Panel actualizaciones** (futuro G14) | Operaciones | UI: versiГіn actual, migraciones pendientes, changelog, ventana mantenimiento |

### PolГӯtica sugerida de versiones

- **Patch (x.y.Z):** fixes, sin migraciГіn destructiva.
- **Minor (x.Y.0):** features, migraciones additive, seeds nuevos.
- **Major (X.0.0):** breaking API; ventana migraciГіn datos + comunicaciГіn analistas.

## 7. FAQ вҖ” Analista de implementaciГіn

### ВҝPuedo agregar un campo a Factura como en SIGACFG sin programador?
SГӯ, si la pantalla ya lee campos personalizados (capa 2). Si no, el analista levanta ticket capa 5 para cablear la UI.

### ВҝLa personalizaciГіn de un cliente afecta a otros?
No, si usa `empresaId` (campos, features, workflows instancia). SГӯ, si se mergea cГіdigo core sin feature flag.

### ВҝCГіmo escalo horizontalmente?
MГЎs instancias de la app + Postgres con connection pooling; Redis para colas (pendiente). No hay вҖңAppServer por clienteвҖқ.

### ВҝCГіmo escalo вҖңverticalmenteвҖқ por rubro?
Activar mГіdulos del rubro (gastronomГӯa vs agro vs industria) vГӯa railroad; no requiere otro binario.

### ВҝActualizo un solo cliente?
En SaaS compartido: **no** (mismo deploy). Opciones: capa 2вҖ“3 por tenant, o instancia dedicada enterprise.

### ВҝQuГ© pasa en un upgrade si el cliente tenГӯa AdvPL custom?
Equivalente aquГӯ: custom capa 5 puede requerir re-merge. Preferir capas 1вҖ“4 para upgrades painless.

### ВҝDГіnde estГЎ el diccionario?
- **Estructural:** `prisma/schema.prisma` (como diccionario SQL central).
- **Por empresa:** tablas `campos_personalizados`, `features_empresa`, `workflow_*`, `tablas_auxiliares`.

### ВҝHay interfaz de parche conectada?
**No hoy.** ActualizaciГіn = deploy + `prisma migrate deploy`. Roadmap: panel G14 (В§5).
