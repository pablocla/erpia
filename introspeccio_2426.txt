### DEUDA TÉCNICA

- plan-cuentas.ts hardcodeado coexiste con CuentaContable DB-backed — duplicación inofensiva pero confusa
- Silent error handling en event handlers — errores se loguean pero no se propagan; un asiento que falla no aborta la factura
- Event handler registration por side-effects de import — frágil, depende del orden de importación
- JWT_SECRET tiene fallback dev inseguro — en producción sin env var, silenciosamente usa key débil
- No hay tests de integración end-to-end (solo unit con mocks)
- prisma db push nunca se ejecutó contra una base real (schema pendiente de deploy)

### DEPENDENCIAS CRÍTICAS FALTANTES

- node-usb o serialport — para I/O con impresoras fiscales USB
- No hay librería de PDF — si se quiere generar factura PDF (alternativa a impresora)
- No hay email service — para envío de facturas por correo
- No hay job scheduler — para tareas periódicas (cierre mensual, alertas stock, CAEA automático)

## 3. ARQUITECTURA ACTUAL vs ERP MULTI-TENANT REAL

### Multi-empresa

| Aspecto | Estado | Detalle |
|---------|--------|---------|
| Modelo Empresa | EXISTE | empresaId en Usuario, Factura, Caja, PuntoVentaConfig, CuentaContable, ConfigAsientoCuenta, Numerador, ParametroFiscal |
| Filtro por empresa en queries | PARCIAL | Algunas queries filtran por empresaId (auth middleware extrae del JWT), pero no todos los endpoints lo hacen consistentemente |
| Aislamiento de datos hard | NO | No hay Row-Level Security (RLS) ni policies en Postgres. Un bug puede devolver datos de otra empresa. |
| Multi-tenant real | NO | Single database, single schema. No hay schema-per-tenant ni database-per-tenant. |

Veredicto: Funciona para demo/MVP con 1 empresa. Para producción multi-cliente necesita: (a) auditar todo endpoint para filtrar por empresaId, (b) considerar RLS en PostgreSQL, (c) eventualmente schema-per-tenant si la escala lo justifica.

### Períodos fiscales

| Aspecto | Estado |
|---------|--------|
| Modelo de período fiscal explícito | NO EXISTE |
| Cierre contable mensual | NO EXISTE |
| Bloqueo de períodos cerrados | NO EXISTE |
| Los queries de IVA/IIBB/SICORE usan mes+año | SÍ, pero no hay lock |

Veredicto: Se puede consultar por período, pero no hay concepto de "período cerrado" que impida modificar asientos/facturas de meses anteriores. Esto es un BLOQUEANTE regulatorio para contabilidad real.

## 4. ESTIMACIÓN HONESTA

### Tiempo hasta producción con un cliente real

| Escenario | Tiempo estimado | Alcance |
|-----------|----------------|---------|
| MVP mínimo (facturar, cobrar, stock, IVA, sin impresora física) | 1-2 semanas | Un comercio que factura por pantalla/PDF, con contabilidad básica |
| POS retail completo (con impresora) | 3-4 semanas | Necesita implementar I/O USB/red para Hasar/Epson |
| Contabilidad completa (períodos fiscales, cierre, IIBB) | 3-4 semanas | Cierre mensual, períodos bloqueables, DDJJ IIBB |
| Multi-empresa seguro (RLS, filtrado consistente) | 2-3 semanas | Auditar todos los endpoints + RLS PostgreSQL |
| ERP full con módulos verticales | 6-8 semanas | Activos fijos, presupuestos→pedidos, multi-moneda real, padrón IIBB |

### Los 3 BLOQUEOS más críticos ahora mismo

1. NO HAY CIERRE DE PERÍODO CONTABLE
Sin un modelo de "período fiscal" con estado abierto/cerrado, cualquier usuario puede crear o modificar asientos de meses pasados. Esto invalida toda la contabilidad para un contador real. Necesita: modelo PeriodoFiscal + validación en AsientoService + check en todos los endpoints que escriben transacciones.

2. IMPRESIÓN FÍSICA ES 100% STUB
Para un POS de mostrador, el cliente necesita imprimir tickets/facturas. Los comandos ESC/POS se generan correctamente (Hasar SMH-P-441, Epson TM-T20) pero enviarComandos() es console.log(). Necesita: librería de comunicación USB (node-usb/serialport) o raw TCP socket para impresoras de red. Alternativa: servicio local tipo CUPS/print-agent que reciba HTTP y rutee a la impresora.

3. AISLAMIENTO MULTI-EMPRESA INCOMPLETO
El JWT incluye empresaId y el middleware lo verifica, pero no todos los endpoints filtran consultas por empresaId. Un endpoint que no filtre puede devolver datos de otra empresa. Necesita: auditoría completa de los 88 route.ts + test de integration que valide aislamiento.

## 5. LO QUE ESTÁ BIEN HECHO — NO TOCAR

### Arquitectura core
- Event bus con ConfiguracionFuncional — Patrón sólido. Handlers activables/desactivables por DB sin redeploy. HandlerLog para auditoría. No cambiar el patrón; solo agregar handlers.
- Tax Engine multi-país (lib/tes/) — Registry pattern con AR/CL/MX adapters. TaxConcepto + TaxTasa en DB con vigencias. Permite cambiar alícuotas sin deploy. Arquitectura limpia.
- ParametroService con cache + fallback — getParametro() con DB → hardcoded default + cache 60s. Pattern correcto para configuración en runtime.
- AFIP SOAP client — WSAA auth con CMS/PKCS#7 signing correcto. WSFEv1 emission probado. Manejo de certificados base64. La crypto funciona.
- AsientoService — Partida doble con validación de balance. 5 tipos de asiento automático + manual. Plan de cuentas DB-backed con mapeo configurable. Sólido.

### Modelo de datos
- Schema Prisma de 115 modelos — Cubre comercio, industria, servicios, salud, hotelería, gastronomía, logística, IoT. No refactorear el schema; solo agregar campos si faltan.
- Maestros paramétricos — 30+ tablas de referencia con CRUD genérico. Provincias con código AFIP, bancos con código BCRA, monedas ISO 4217. Seed de 200+ registros. Data foundation correcta.
- Soft-delete (deletedAt) — Patrón consistente en modelos core. No cambiar a hard-delete.

### RBAC
- 9 roles con permisos granulares — Módulos permitidos, descuento máximo, acceso web/mobile, visibilidad de costos/sueldos. Diseño profesional tipo Odoo/Protheus.

### UI
- 37 páginas dashboard con implementación real — No son stubs. Cada módulo tiene UI funcional con formularios, tablas, filtros, exports.
- Onboarding wizard paso a paso — La experiencia de usuario para dar de alta una empresa nueva está diseñada. Solo falta la lógica de recomendación backend.

## RESUMEN EJECUTIVO

Este no es un demo. Es un ERP argentino funcional con 115 modelos, 88 API routes y 37 páginas de UI reales.

El 85% del backend está implementado con lógica de negocio real. Los flujos core (facturar con AFIP → stock → contabilidad → CC → cobro) funcionan end-to-end via events.

Para llegar a producción con un primer cliente:

1. Implementar PeriodoFiscal + cierre contable
2. Implementar IIBB cierre mensual + DDJJ
3. Auditar filtro empresaId en los 88 endpoints
4. Decidir: impresora física (USB/red) vs PDF/pantalla
5. prisma db push a una base PostgreSQL real + seed data
6. Correr el onboarding con datos de un cliente real

El sistema está diseñado para que un implementador entre, elija rubro, y tenga plan de cuentas + módulos + parámetros fiscales configurados automáticamente. La base está. Falta cerrar los gaps operativos para que un contador lo firme.
