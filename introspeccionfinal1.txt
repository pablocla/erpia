PARTE 2 — LO QUE NOS FALTA (CRÍTICO, SIN EXCUSAS)

────────────────────────────────────────────────────────────
2.1  INFRAESTRUCTURA — BLOQUEADORES DE PRODUCCIÓN
────────────────────────────────────────────────────────────

🔴 SIN COLA DE MENSAJES / BACKGROUND JOBS
   Problema: Los cron jobs son endpoints HTTP normales de Next.js.
   Si la ejecución de 10 agentes para 100 empresas toma 50 minutos,
   el siguiente cron puede solaparse. No hay retry en falla.
   No hay dead letter queue para jobs fallidos.
   No hay separación worker/web — todo corre en el mismo proceso.
   Impacto: A 30+ clientes, los agentes van a fallar silenciosamente.
   Solución: BullMQ + Redis (o Inngest/Trigger.dev serverless)

🔴 SIN REDIS
   Problema: Rate limiting es in-memory (reset en cada deploy).
   Event bus es in-memory (eventos en vuelo se pierden en restart).
   Concurrency guard de agentes es in-memory (multi-instancia = doble ejecución).
   Cache de AI context (60s TTL) es in-memory (cada instancia repite queries).
   Impacto: En multi-instancia (Railway auto-scale, Vercel edge), TODO se rompe.
   Solución: Redis/Upstash para: rate limit, locks, cache, pub/sub, queues

🔴 SIN DISTRIBUTED LOCKS
   Problema: 2 instancias pueden ejecutar el mismo agente al mismo tiempo.
   El check de "agenteLog reciente" tiene race condition:
     T=0ms: Instancia A lee agenteLog → no encuentra nada
     T=0ms: Instancia B lee agenteLog → no encuentra nada
     T=1ms: Ambas ejecutan → 2 alertas duplicadas
   Mismo problema con asientos contables en event handlers.
   Solución: Postgres advisory locks (SELECT pg_advisory_xact_lock(hash))
   o Redis SETNX para distributed locking

🔴 SIN OBSERVABILIDAD / MONITORING
   Problema: Solo console.log/console.error. En Railway los logs se pierden.
   No hay trazas de performance. No hay alertas en errores 500.
   No hay métricas de latencia de AI, duración de queries, throughput.
   No hay health checks de Ollama/Anthropic con circuit breaker.
   Impacto: Cosas van a fallar en producción y no nos vamos a enterar.
   Solución: Sentry (errores) + OpenTelemetry/Datadog (trazas) + Better Stack (logs)

🔴 SIN WEBSOCKET / SSE (REAL-TIME)
   Problema: El dashboard no puede recibir notificaciones push.
   Cuando un agente termina de ejecutar, el usuario no se entera.
   Cuando entra una alerta de stock crítico, no aparece hasta refresh.
   El KDS de cocina tiene que hacer polling cada 5 segundos.
   Impacto: UX inferior a cualquier SaaS moderno. Gastronomía necesita real-time YA.
   Solución: Server-Sent Events (SSE) mínimo para notificaciones.
   Socket.io/Pusher si se necesita bidireccional.

🟡 SIN FILE UPLOAD SYSTEM
   Problema: Certificados AFIP guardados como Base64 en la DB.
   No hay soporte para subir logos, fotos de evidencia de entrega,
   fotos de cheques, imágenes de productos.
   Impacto: 10MB+ de archivos por empresa van a inflar la DB.
   Solución: S3/Cloudflare R2 presigned URLs + campo URL en modelo.

🟡 SIN WEBHOOKS SALIENTES
   Problema: No podemos notificar a sistemas externos cuando pasan cosas.
   Ejemplo: "Cuando factura > $500K, notificar a Slack del contador".
   Ejemplo: "Cuando stock llega a 0, enviar webhook a e-commerce para marcar sin stock".
   Impacto: No hay integraciones con terceros.
   Solución: Tabla WebhookConfig + WebhookDelivery con retry exponencial.

🟡 SIN API VERSIONING
   Problema: Todas las rutas son /api/clientes, /api/facturas sin /v1/.
   Un cambio breaking requiere big-bang migration de todos los clientes.
   Impacto: Cuando tengamos 50+ clientes, un cambio en la API rompe todo.

────────────────────────────────────────────────────────────
2.2  AGENTES IA — GAPS FUNCIONALES
────────────────────────────────────────────────────────────

🔴 NO HAY IDEMPOTENCIA EN AGENTES
   Problema: Si el cron se ejecuta 2 veces (retry, deploy, race condition),
   se crean alertas duplicadas. No hay constraint de unicidad.
   Código: prisma.alertaIA.create() sin check de existencia previa.
   Solución: UNIQUE constraint (empresaId, agenteId, DATE_TRUNC('day', createdAt))
   o idempotency key en cada acción.

🔴 NO HAY RETRY CON BACKOFF EXPONENCIAL
   Problema: Si Ollama está lento o Anthropic da 429 (rate limit),
   el agente falla con error permanente. No reintenta.
   Impacto: En horario pico de uso de GPU, 50%+ de agentes pueden fallar.
   Solución: 3 reintentos con backoff (1s, 4s, 16s) antes de marcar error.

🔴 NO HAY CIRCUIT BREAKER PARA AI
   Problema: Si Ollama tarda 119 segundos en cada request, el sistema
   quema el timeout completo en cada agente antes de caer a Anthropic.
   Para 10 agentes × 100 empresas = 33 HORAS de espera inútil.
   Solución: Circuit breaker: si Ollama falla 3 veces seguidas,
   abrir circuito y saltar directo a Anthropic por 5 minutos.

🟡 NO HAY FEEDBACK LOOP
   Problema: Las alertas se generan pero nadie marca "fue útil/fue basura".
   Sin feedback, la IA no puede mejorar. Siempre genera las mismas alertas.
   Solución: Campo "feedback" en AlertaIA (util/no_util/falso_positivo)
   + usar feedback como contexto en el próximo ciclo.

🟡 NO HAY PROMPT CACHING
   Problema: Cada ejecución de agente re-envía el contexto completo de la empresa
   (25 queries Prisma paralelas, reconstruyen todo). A 100 empresas × 10 agentes
   = 25,000 queries por ciclo nocturno. Con cache central duraría 2,500.
   Solución: Redis cache del contexto de empresa con TTL de 5 minutos.

🟡 WHATSAPP SENDER NO EXISTE
   Problema: Toda la infraestructura de mensajes WhatsApp está lista —
   tabla de mensajes pendientes, UI de aprobación, generadores IA,
   cron de envío — pero el sender real es console.log.
   Impacto: La línea de negocio "Agente de Cobranzas por WhatsApp" no funciona.
   Solución: Twilio o 360dialog. ~20 líneas de código real.

────────────────────────────────────────────────────────────
2.3  TESTING — COBERTURA INSUFICIENTE
────────────────────────────────────────────────────────────

🟡 SIN TESTS E2E
   Problema: 111 tests unitarios cubren ~40% de la lógica.
   No hay tests que hagan: login → crear cliente → crear factura → cobrar → verificar asiento.
   Solución: Playwright o Cypress para flujos críticos (5-10 tests E2E).

🟡 SIN TESTS DE AUTH
   Problema: El middleware JWT no tiene tests. Si alguien refactoriza
   y rompe la validación, no hay test que lo detecte.

🟡 SIN TESTS DE INTEGRACIÓN CON DB REAL
   Problema: Todos los tests mockean Prisma. No verifican constraints reales,
   triggers, indexes, o race conditions de la DB.
   Solución: Test container con PostgreSQL real para integration tests.

────────────────────────────────────────────────────────────
2.4  FUNCIONAL — GAPS DE NEGOCIO
────────────────────────────────────────────────────────────

🟡 SIN FACTURA ELECTRÓNICA DE EXPORTACIÓN (tipo E)
   Argentina exige para exportadores. Bajo demanda pero necesario.

🟡 SIN LIBRO IVA DIGITAL
   RG 4597/2019 — obligatorio para Responsable Inscripto.
   Tenemos los datos (tributos por factura/compra) pero no el exportador.

🟡 SIN INTEGRACIÓN MERCADO LIBRE
   Prometida en pricing. Necesita ML API + sync de stock.

🟡 SIN REPORTING / BI
   No hay reportes configurables por el usuario. Solo los que genera la IA.
   Falta: tabla dinámica, gráficos, export a Excel con formato.

🟡 SIN MULTI-MONEDA OPERATIVA
   Hay CotizacionService y campo moneda en factura, pero no hay
   flujo real de operar en USD y contabilizar en ARS.

PARTE 3 — QUÉ IMPLEMENTAR PARA DIFERENCIACIÓN MÁXIMA 2026
═══════════════════════════════════════════════════════════════════════════

Acá la pregunta real: con lo que tenemos (~65% construido), ¿qué agregaríamos
para que NINGÚN competidor en Argentina/LATAM pueda ni acercarse?

────────────────────────────────────────────────────────────
3.1  INFRA QUE ESCALA — "THE BORING STUFF THAT WINS"
────────────────────────────────────────────────────────────

PRIORIDAD 1: REDIS + BULLMQ (Semana 1)
  • Redis en Upstash (serverless, $0 hasta 10K reqs/día, gratis para empezar)
  • BullMQ para todos los jobs: agentes IA, envío de emails, envío WhatsApp,
    generación de PDF, sync AFIP
  • Workers separados del proceso web (Railway worker service)
  • Dashboard de BullMQ: ver jobs pendientes, fallidos, completados
  • Retry automático con backoff exponencial (3 intentos, 1s/4s/16s)
  • Dead letter queue para jobs que fallaron 3 veces → alerta al admin
  ¿Por qué diferencia? Ningún ERP PyME en Argentina tiene procesamiento
  asíncrono real. Bind, Colppy, Xubio — todos bloquean al usuario.

PRIORIDAD 2: DISTRIBUTED LOCKS (Semana 1, junto con Redis)
  • Postgres advisory locks en agentes: SELECT pg_advisory_xact_lock(hash)
  • O Redis SETNX con TTL para lock distribuido
  • Idempotency keys en cada acción de agente
  • Constraint UNIQUE en AlertaIA para evitar duplicados
  ¿Por qué diferencia? Podemos escalar a N instancias sin bugs.

PRIORIDAD 3: OBSERVABILIDAD (Semana 2)
  • Sentry para error tracking (free tier: 5K events/mes)
  • OpenTelemetry para trazas distribuidas
  • Structured logging con JSON (pino) → Better Stack o Datadog
  • Health check mejorado: /api/health que verifica DB + Redis + Ollama + AFIP
  • Dashboard de operaciones: latencia por endpoint, errores/hora, uso de GPU
  ¿Por qué diferencia? Podemos garantizar 99.9% uptime con datos reales.

PRIORIDAD 4: SSE / REAL-TIME (Semana 2-3)
  • Server-Sent Events endpoint: /api/sse/[empresaId]
  • Tipos de evento: alerta_nueva, agente_completado, comanda_nueva,
    stock_critico, factura_aprobada, pago_recibido
  • Toast notifications en el dashboard en real-time
  • KDS de cocina con SSE (elimina polling cada 5s)
  • Notificación a vendedor cuando su pedido cambió de estado
  ¿Por qué diferencia? La experiencia pasa de "sitio web" a "app viva".
  Gastronomía SIN real-time NO ES VIABLE. El KDS necesita 0 latencia.

────────────────────────────────────────────────────────────
3.2  AGENTES IA — DE "EJECUTOR" A "CEREBRO AUTÓNOMO"
────────────────────────────────────────────────────────────

FEATURE: CIRCUIT BREAKER + RETRY INTELIGENTE
  Hoy: Ollama falla → error permanente.
  Mañana: Ollama falla → retry 3x con backoff → circuit open → Anthropic directo → 
          después de 5min → half-open → intenta Ollama de nuevo.
  Implementación: ~60 líneas. Estado: closed/open/half-open.
  Beneficio: 99% de uptime de IA vs 70% actual.

FEATURE: AGENT FEEDBACK LOOP
  Hoy: IA genera alertas, nadie dice si sirvieron.
  Mañana: 
  • Botón "👍 Útil / 👎 No útil / ⚠️ Falso positivo" en cada alerta
  • Los feedback se agregan al prompt del próximo ciclo:
    "En los últimos 7 días, el usuario marcó 3 alertas de stock como falso positivo
     porque los productos tenían OC pendiente. Considerá OC pendientes antes de alertar."
  • Métrica: % de alertas útiles por agente (KPI para el cliente)
  Beneficio: La IA MEJORA sola con el uso. Nadie tiene esto.

FEATURE: AGENT CHAINS (AGENTES QUE LLAMAN A AGENTES)
  Hoy: Cada agente ejecuta independiente.
  Mañana:
  • Agente anomalías detecta precio inusual →
    Trigger automático: agente de auditoría genera reporte →
    Trigger automático: agente de notificaciones manda WhatsApp al dueño
  • Agente de stock detecta producto agotado →
    Trigger: agente compras genera OC sugerida →
    Trigger: agente marketing pausa campaña de ese producto
  Implementación: campo "chainsTo" en AgentConfig + executeChain() en registry.
  Beneficio: AUTOMATIZACIÓN TOTAL sin intervención humana.
  Esto es lo más cercano a un SAP con workflow que existe para PyMEs.

FEATURE: SCHEDULED AGENT PER EMPRESA (CRON PERSONALIZADO)
  Hoy: Todos los agentes corren en los mismos horarios para todas las empresas.
  Mañana:
  • Tabla `ConfiguracionAgente` (empresaId, agenteId, cronExpression, params)
  • Cada empresa elige cuándo corren sus agentes
  • Parámetros por empresa: "alertar si stock < 20" vs "alertar si stock < 5"
  • UI en dashboard: cron builder visual (seleccionar día/hora)
  Beneficio: Cada cliente siente que tiene un sistema PERSONALIZADO.

FEATURE: NATURAL LANGUAGE AGENT TRIGGER
  Hoy: Los agentes se disparan por cron o evento.
  Mañana: El dueño escribe en el chat:
    "Mandales un WhatsApp a los que me deben más de $100K"
    → El sistema interpreta → ejecuta agente de cobranzas con filtro monto > 100K
    → Genera mensajes → Muestra preview → Envía con un click
  Implementación: function-calling / tool-use con los agentes como herramientas.
  Beneficio: INTERFAZ CONVERSACIONAL para operar el ERP completo.
  El dueño del kiosco no necesita saber qué botón apretar.

FEATURE: AI PLAYGROUND / EVALUATION
  Hoy: No sabemos si la IA genera buenas respuestas.
  Mañana:
  • Endpoint /api/ai/eval que corre N prompts predefinidos y compara respuestas
  • Scoring automático: coherencia, relevancia, formato correcto
  • A/B testing de prompts: "prompt v1 vs v2, cuál genera mejores alertas"
  • Dashboard de calidad de IA: accuracy por agente, drift detection
  Beneficio: Podemos medir y GARANTIZAR calidad de IA. SAP no puede.
