## 1. HAMBURGUESERÍA / DARK KITCHEN / FAST FOOD (pedido web + WhatsApp)

**Gap crítico:** Hoy la comanda exige `mesaId` obligatorio (ver route.ts). No existe flujo de pedido sin mesa (delivery/takeaway/web). No hay webhook WhatsApp. No hay tracking de estado para el cliente.

**Ingeniería funcional necesaria:**

| Componente | Descripción |
|-----------|-------------|
| `PedidoOnline` | Modelo nuevo: pedido sin mesa, canal (web/whatsapp/rappi), estado (recibido→confirmado→en_preparacion→listo→en_camino→entregado), dirección delivery, teléfono, pago |
| `WebhookWhatsApp` | Endpoint receptor de mensajes de WhatsApp Business API (Baileys/Twilio/360dialog). Parsea texto natural → items del menú vía fuzzy match |
| `ComandaAutoGenerator` | Service que al recibir PedidoOnline genera Comanda automática sin mesa (mesa virtual "DELIVERY" / "TAKEAWAY") y la envía al KDS |
| `MenuDigital` | API pública (sin auth) que expone catálogo con fotos, precios, variantes (con cheddar, sin cebolla), horarios disponibilidad |
| `EstimadorTiempo` | Calcula tiempo estimado basado en cola actual de comandas abiertas × tiempo promedio |
| `NotificadorCliente` | Envía WhatsApp/SMS al cliente en cada cambio de estado: "Tu pedido está en preparación", "Tu hamburguesa está lista para retirar" |
| `ZonaDelivery` | Polígonos/radios configurables con costo de envío variable |
| `IntegracionRappi/PedidosYa` | Webhook receptor que normaliza pedidos de aggregators al formato PedidoOnline |

**Flujos:**
```
Cliente → Web/WhatsApp → PedidoOnline (borrador)
  → Confirmación pago (MP/efectivo) → ComandaAutoGenerator → Comanda (sin mesa)
  → KDS cocina → Cambio estado → NotificadorCliente WhatsApp
  → Entrega → Facturación automática (FC/B/C)
```

## Resumen de gaps transversales que habilitan TODOS estos nichos

| Servicio transversal | Estado actual | Impacto |
|---------------------|---------------|---------|
| **WhatsApp Business API** | Ausente | Bloquea notificaciones, booking, y canal de pedidos para TODOS los rubros |
| **Booking/Reservas Online** (endpoint público sin auth) | Ausente | Peluquerías, canchas, clínicas, lavaderos, clases — todos lo necesitan |
| **Programa de Fidelidad/Puntos** | Mencionado pero sin modelo ni API | Diferenciador competitivo para retail recurrente |
| **Suscripciones/Recurrencia** | `Membresia` solo tiene fecha inicio/fin | Falta facturación recurrente automática, reintentos, congelamiento |
| **Pedido sin mesa (delivery/take-away/web)** | `Comanda` exige `mesaId` Int obligatorio | Bloquea dark kitchens, delivery, fast food completo |
| **Servicio como entidad separada de Producto** | Ausente — todo es `Producto` | Peluquerías, clínicas, talleres necesitan duración, insumos consumidos, comisión |
| **Seña/Pago parcial anticipado** | Ausente | Tatuajes, alquileres, reservas de cancha, eventos |
| **Consentimiento informado digital** | Ausente | Salud, estética, tatuajes — obligatorio legalmente |
| **Calendario público por recurso** (cancha, profesional, bahía) | Solo `Turno` interno | Falta exposición pública para self-service del cliente |
| **MercadoPago integración** | Mencionado pero ausente | Cobro online para todos los canales web/WhatsApp |
| **Foto/Media por transacción** | Ausente | Tatuajes, lavaderos, veterinaria, peluquerías |

Estos 15 nichos + los 6 servicios transversales son la hoja de ruta para que el ERP pase de "genérico con verticales cosméticos" a **plataforma multi-rubro real** que compite con Fresha (belleza), Toast (gastro), Mindbody (fitness), y Odoo (genérico) — pero con fiscalidad argentina nativa.
